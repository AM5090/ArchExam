# Задание 3. Анализ архитектуры учебной информационной системы

## Что архитектурно неудачно
> Нарушен принцип разделения уровней (Separation of Concerns) — бизнес-логика (скидки, льготы, обработка заявок) напрямую смешана с инфраструктурными деталями:

- HTTP-слой (контроллеры)
- ORM и SQL-запросы
- Конкретные механизмы хранения данных

### Последствия:

- Бизнес-правила невозможно проверить без базы данных или поднятого веб-контейнера
- Смена БД или API требует переписывания бизнес-логики
- Высокая связность, низкая тестируемость
- Нарушение Open/Closed Principle и Dependency Inversion Principle

## Какой подход уместен
> Предметно-ориентированное проектирование (Domain-Driven Design, DDD) с выделением доменного слоя (Domain Layer).

### Конкретные элементы подхода:

- ```Ядро домена (Domain Model)``` — бизнес-правила в чистом виде (POJO), без зависимостей на инфраструктуру
- ```Антикоррупционный слой (Anti-Corruption Layer)``` — при работе с внешними API
- ```Хранилища (Repository pattern)``` — абстракция доступа к данным на уровне интерфейсов в домене
- ```Сервисы приложения (Application Services)``` — координация между доменом и инфраструктурой

## За счет чего решается проблема

| Проблема | Как решается |
| :--- | :--- |
| Бизнес-логика завязана на HTTP-слой | Контроллеры становятся тонкими — только принимают запрос, вызывают Application Service, возвращают ответ. Бизнес-правила вынесены в Domain Layer |
| Бизнес-логика завязана на ORM/SQL | Вводится Repository interface в доменном слое, а реализация с ORM — в инфраструктурном. Домен не знает, как хранятся данные |
| Смена БД требует правки бизнес-логики | Домен зависит от абстракции (интерфейса репозитория), а не от конкретной БД. Меняется только реализация в инфраструктуре |
| Сложно тестировать бизнес-правила | Чистый доменный слой тестируется без базы данных и HTTP (обычные unit-тесты) |
| Изменение внешнего API ломает бизнес-логику | Внешние API изолируются за Anti-Corruption Layer — бизнес-логика работает с собственными моделями, адаптер переводит их в формат внешней системы |

## Краткий вывод
Архитектурно неудачно отсутствие четкого доменного слоя и смешение бизнес-правил с инфраструктурой. Уместен подход DDD с выделением чистого доменного ядра, независимого от внешних деталей. Это достигается за счет инверсии зависимостей (Dependency Inversion) — бизнес-логика зависит только от абстракций (интерфейсов), а конкретные реализации (БД, HTTP, API) подключаются на периферии.
