# Задание 4. Повышение наблюдаемости системы

#### Какие проблемы выявлены:

1. Логи хранятся локально внутри контейнеров и теряются при перезапуске
2. Нет единой картины по метрикам
3. Нет единой картины по логам
4. О проблемах узнают от пользователей, а не из мониторинга
5. Причины роста времени ответа не анализируются автоматически

## Предлагаемые инструменты и практики

### 1. Централизованный сбор и хранение логов

```Инструменты```: ELK Stack (Elasticsearch + Logstash + Filebeat) или Loki + Grafana

```Как работает```: Filebeat / Promtail собирает логи с каждого контейнера и отправляет в центральное хранилище (Elasticsearch / Loki)

#### Какая проблема решается:
- Логи не теряются при перезапуске контейнера
- Логи доступны из одного места для всей команды
- Возможен поиск по всем контейнерам сразу

### 2. Сбор метрик и мониторинг

```Инструменты```: Prometheus + Grafana

```Как работает```: Prometheus собирает метрики с экспортеров (cAdvisor для контейнеров, Node exporter для узлов, приложение отдает метрики на /metrics)

#### Ключевые метрики:

- Время ответа
- RPS
- Количество ошибок
- Потребление CPU / RAM по каждому контейнеру
- Количество перезапусков контейнеров

#### Какая проблема решается:
- Команда видит рост времени ответа на графиках раньше жалоб пользователей
- Видна нагрузка на узлы и контейнеры
- Перезапуски контейнеров видны как события на дашборде

### 3. Оповещение

```Инструменты```: Alertmanager (часть Prometheus)

```Как работает```: Настраиваются правила — при превышении порогов (например, p95 latency > 500ms или контейнер перезапустился > 3 раз за 5 минут) отправляется уведомление в Telegram / Slack / PagerDuty

#### Какая проблема решается:
- Команда узнает о проблеме до пользователей
- Автоматическое оповещение вместо ручного мониторинга дашбордов

### Краткий вывод

Для повышения наблюдаемости необходимо внедрить:

```Логи``` — централизованные (ELK/Loki)
```Метрики``` — сбор и визуализация (Prometheus + Grafana)

Дополнительно — аллертинг для опережающего оповещения и для контроля перезапусков. Это позволит команде видеть проблемы раньше пользователей и сократит время разбора инцидентов.
